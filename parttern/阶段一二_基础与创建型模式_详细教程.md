## 1. 设计模式概述

### 1.1 什么是设计模式

设计模式是针对软件设计中**反复出现的问题**提出的通用解决方案。它不是代码，而是一套经过验证的**设计经验总结**。

**历史背景**：1994 年，四位作者（Erich Gamma、Richard Helm、Ralph Johnson、John Vlissides）合著了《设计模式：可复用面向对象软件的基础》，被称为 **GoF（Gang of Four，四人帮）**。书中总结了 23 种经典设计模式。

### 1.2 设计模式的核心价值

- **复用解决方案**：不必重复造轮子，直接应用经过验证的设计
- **促进团队沟通**：设计模式是通用词汇，团队成员可以用模式名称快速对齐思路
- **提升代码质量**：遵循模式可以让代码更易维护、扩展

### 1.3 三大分类

| 分类 | 核心问题 | 模式数量 |
|------|---------|---------|
| **创建型** | 对象如何创建，将创建与使用分离 | 5 种 |
| **结构型** | 类/对象如何组合成更大结构 | 7 种 |
| **行为型** | 对象间的职责分配与算法协作 | 11 种 |

## 2. SOLID 面向对象原则

> SOLID 是 5 个最重要设计原则的首字母，是所有设计模式的理论根基。

### 2.1 SRP — 单一职责原则（Single Responsibility Principle）

**核心**：一个类只做一件事，只有一件引起它变化的原因。

```csharp
// 违反 SRP：UserService 既处理业务又处理日志和邮件
public class UserService
{
    public void Register(User user)
    {
        // 注册逻辑
        Console.WriteLine("User registered");
        // 日志逻辑
        Console.WriteLine("Log: User registered");
        // 邮件逻辑
        Console.WriteLine("Email sent");
    }
}

// 符合 SRP：职责分离
public class UserService
{
    private readonly ILogger _logger;
    private readonly IEmailService _emailService;

    public UserService(ILogger logger, IEmailService emailService)
    {
        _logger = logger;
        _emailService = emailService;
    }

    public void Register(User user)
    {
        // 仅处理注册业务
        _logger.Info("User registered");
        _emailService.SendWelcome(user);
    }
}
```

### 2.2 OCP — 开闭原则（Open-Closed Principle）

**核心**：对扩展开放，对修改关闭。软件实体应该通过扩展来实现变化，而不是修改已有代码。

```csharp
// 违反 OCP：新增折扣类型必须修改 Order 类
public class Order
{
    public decimal CalculateDiscount(string discountType)
    {
        if (discountType == "VIP") return 0.8m;
        if (discountType == "Seasonal") return 0.9m;
        // 新折扣类型必须修改这里
        return 1.0m;
    }
}

// 符合 OCP：通过扩展而非修改实现新折扣
public interface IDiscountStrategy
{
    decimal GetDiscount(Order order);
}

public class NoDiscount : IDiscountStrategy
{
    public decimal GetDiscount(Order order) => 1.0m;
}

public class VipDiscount : IDiscountStrategy
{
    public decimal GetDiscount(Order order) => 0.8m;
}

public class SeasonalDiscount : IDiscountStrategy
{
    public decimal GetDiscount(Order order) => 0.9m;
}

// 新增折扣只需添加新类，无需修改 Order
```

### 2.3 LSP — 里氏替换原则（Liskov Substitution Principle）

**核心**：子类必须能够替换其父类。使用父类的地方，换成子类不会出错。

```csharp
// 违反 LSP：Rectangle 的宽度和高度可以是不同的值，
// 但 Square 强制它们相等，导致调用方行为异常
public class Rectangle
{
    public virtual double Width { get; set; }
    public virtual double Height { get; set; }

    public double Area() => Width * Height;
}

public class Square : Rectangle
{
    public override double Width
    {
        get => base.Width;
        set { base.Width = value; base.Height = value; }
    }

    public override double Height
    {
        get => base.Height;
        set { base.Width = value; base.Height = value; }
    }
}

// 测试代码期望 Rectangle，但 Square 行为不一致
void TestArea(Rectangle r)
{
    r.Width = 5;
    r.Height = 4;
    Debug.Assert(r.Area() == 20); // Square 会失败：Area = 16
}
```

### 2.4 ISP — 接口隔离原则（Interface Segregation Principle）

**核心**：接口要小而专，不要强迫客户端依赖不用的方法。

```csharp
// 违反 ISP：一个臃肿的 IMachine 接口
public interface IMachine
{
    void Print(Document d);
    void Scan(Document d);
    void Fax(Document d);
}

// 老式打印机只实现 Print，其他方法需抛异常或空实现
public class OldPrinter : IMachine
{
    public void Print(Document d) { /* 打印 */ }
    public void Scan(Document d) { throw new NotImplementedException(); }
    public void Fax(Document d) { throw new NotImplementedException(); }
}

// 符合 ISP：拆分为小接口
public interface IPrinter { void Print(Document d); }
public interface IScanner { void Scan(Document d); }
public interface IFax { void Fax(Document d); }

public class OldPrinter : IPrinter
{
    public void Print(Document d) { /* 打印 */ }
}

public class ModernMachine : IPrinter, IScanner, IFax
{
    public void Print(Document d) { }
    public void Scan(Document d) { }
    public void Fax(Document d) { }
}
```

### 2.5 DIP — 依赖倒置原则（Dependency Inversion Principle）

**核心**：高层模块不要依赖低层模块，两者都应该依赖抽象。抽象不要依赖细节，细节应该依赖抽象。

```csharp
// 违反 DIP：HighLevel 直接依赖 LowLevel
public class MySQLDatabase
{
    public void Save(string data) { /* MySQL 保存 */ }
}

public class BusinessLogic
{
    private readonly MySQLDatabase _db = new MySQLDatabase(); // 强耦合

    public void Save(string data) => _db.Save(data);
}

// 符合 DIP：依赖抽象（接口）
public interface IRepository
{
    void Save(string data);
}

public class MySQLRepository : IRepository
{
    public void Save(string data) { /* MySQL */ }
}

public class MongoDBRepository : IRepository
{
    public void Save(string data) { /* MongoDB */ }
}

public class BusinessLogic
{
    private readonly IRepository _repository;

    // 通过注入获得具体实现，高层不依赖低层
    public BusinessLogic(IRepository repository)
    {
        _repository = repository;
    }

    public void Save(string data) => _repository.Save(data);
}
```

### 2.6 补充原则

| 原则 | 全称 | 核心含义 |
|------|------|---------|
| DRY | Don't Repeat Yourself | 不要重复自己，消除重复代码 |
| KISS | Keep It Simple, Stupid | 保持简单，不要过度设计 |
| LOD | Law of Demeter | 迪米特法则，只与直接朋友通信 |

### 3.1 Singleton — 单例模式

#### 模式意图

确保一个类只有一个实例，并提供一个全局访问点。

#### 常见应用场景

- 全局配置管理器
- 数据库连接池
- 日志记录器
- 线程池管理

#### 核心实现方式

**方式一：饿汉式（类加载时即创建）**

```csharp
public sealed class SingletonEager
{
    // 类加载时直接创建实例，线程安全（JLS 保证）
    private static readonly SingletonEager _instance = new SingletonEager();

    private SingletonEager() { }

    public static SingletonEager Instance => _instance;
}
```

**方式二：懒汉式 + 双重检查锁（推荐）**

```csharp
public sealed class SingletonLazy
{
    // volatile 防止指令重排序，确保 instance 引用指向完整构造对象
    private static volatile SingletonLazy _instance;
    private static readonly object _lock = new object();

    private SingletonLazy() { }

    public static SingletonLazy Instance
    {
        get
        {
            if (_instance == null) // 第一次检查：避免不必要的锁开销
            {
                lock (_lock)
                {
                    if (_instance == null) // 第二次检查：确保只创建一个实例
                    {
                        _instance = new SingletonLazy();
                    }
                }
            }
            return _instance;
        }
    }
}
```

**方式三：静态内部类（最简洁的线程安全写法）**

```csharp
public sealed class SingletonNested
{
    private SingletonNested() { }

    // 延迟初始化：只有访问 Instance 时才加载 Nested 类
    private static class Nested
    {
        // C# 规范保证字段初始化和类型加载是线程安全的
        internal static readonly SingletonNested Instance = new SingletonNested();
    }

    public static SingletonNested Instance => Nested.Instance;
}
```

**方式四：Lazy<T>（C# 内置，简洁安全）**

```csharp
public sealed class SingletonLazyBuiltIn
{
    // Lazy<T> 本身就是线程安全的，无需手动加锁
    private static readonly Lazy<SingletonLazyBuiltIn> _instance =
        new Lazy<SingletonLazyBuiltIn>(() => new SingletonLazyBuiltIn());

    private SingletonLazyBuiltIn() { }

    public static SingletonLazyBuiltIn Instance => _instance.Value;
}
```

#### 单例模式的优缺点

| 优点 | 缺点 |
|------|------|
| 全局唯一实例，节省资源 | 职责过重，违反 SRP |
| 全局访问点，方便协调 | 难以进行单元测试 |
| 懒加载可控 | 与依赖注入思想冲突 |

### 3.2 Factory Method — 工厂方法模式

#### 模式意图

定义一个创建对象的接口，但由子类决定实例化哪个类。工厂方法使一个类的实例化延迟到其子类。

#### 结构

```
Creator（创建者）
  └─ FactoryMethod() → Product

ConcreteCreator（具体创建者）
  └─ FactoryMethod() → ConcreteProduct

Product（产品接口）
  └─ Operation()

ConcreteProduct（具体产品）
  └─ Operation()
```

#### 应用场景

- 日志记录器创建（文件日志、数据库日志、控制台日志）
- 数据库连接创建（SQL Server、MySQL、PostgreSQL）
- 文档格式处理（Word、PDF、HTML）

#### 代码实现

```csharp
// ==================== 产品层 ====================

/// <summary>
/// 产品接口：所有物流运输方式的抽象
/// </summary>
public interface ITransport
{
    string Deliver();
}

/// <summary>
/// 具体产品：卡车运输
/// </summary>
public class TruckTransport : ITransport
{
    public string Deliver() => "Deliver by land in a truck";
}

/// <summary>
/// 具体产品：轮船运输
/// </summary>
public class ShipTransport : ITransport
{
    public string Deliver() => "Deliver by sea in a ship";
}

/// <summary>
/// 具体产品：飞机运输
/// </summary>
public class AirplaneTransport : ITransport
{
    public string Deliver() => "Deliver by air in an airplane";
}

// ==================== 工厂层 ====================

/// <summary>
/// 创建者：定义工厂方法，返回产品接口
/// </summary>
public abstract class Logistics
{
    // 工厂方法：子类决定创建哪个产品
    protected abstract ITransport CreateTransport();

    // 业务逻辑：使用产品，但不关心具体是哪种
    public string PlanDelivery()
    {
        ITransport transport = CreateTransport();
        return transport.Deliver();
    }
}

/// <summary>
/// 具体创建者：公路物流
/// </summary>
public class RoadLogistics : Logistics
{
    protected override ITransport CreateTransport() => new TruckTransport();
}

/// <summary>
/// 具体创建者：海上物流
/// </summary>
public class SeaLogistics : Logistics
{
    protected override ITransport CreateTransport() => new ShipTransport();
}

/// <summary>
/// 具体创建者：航空物流
/// </summary>
public class AirLogistics : Logistics
{
    protected override ITransport CreateTransport() => new AirplaneTransport();
}

// ==================== 客户端代码 ====================

class Program
{
    static void Main()
    {
        // 客户端通过创建不同的 Logistics 子类，获得不同的运输方式
        Logistics logistics = new RoadLogistics();
        Console.WriteLine(logistics.PlanDelivery()); // Deliver by land in a truck

        logistics = new SeaLogistics();
        Console.WriteLine(logistics.PlanDelivery()); // Deliver by sea in a ship

        logistics = new AirLogistics();
        Console.WriteLine(logistics.PlanDelivery()); // Deliver by air in an airplane
    }
}
```

#### 工厂方法 vs 简单工厂

| 维度 | 简单工厂 | 工厂方法 |
|------|---------|---------|
| 扩展性 | 新产品需修改工厂类 | 新产品只需添加新工厂类 |
| OCP 符合度 | 部分违反 | 完全符合 |
| 复杂度 | 简单 | 稍复杂 |

### 3.3 Abstract Factory — 抽象工厂模式

#### 模式意图

提供一个创建**一系列相关对象**的接口，而无需指定它们的具体类。

#### 结构

```
AbstractFactory     — 创建抽象产品
    └─ CreateProductA() → AbstractProductA
    └─ CreateProductB() → AbstractProductB

ConcreteFactory1    — 创建产品族 1
    └─ CreateProductA() → ProductA1
    └─ CreateProductB() → ProductB1

ConcreteFactory2    — 创建产品族 2
    └─ CreateProductA() → ProductA2
    └─ CreateProductB() → ProductB2

Client → 只使用 AbstractFactory 和 AbstractProduct，不依赖具体类
```

#### 应用场景

- **跨平台 UI 组件**：同一主题的皮肤组件必须配套使用
- **数据库访问层**：同一事务内的多个表操作需使用同一数据库连接
- **游戏主题资源**：同一游戏主题的道具、角色、特效应协调

#### 代码实现

```csharp
// ==================== 产品族 A：按钮 ====================

public interface IButton
{
    void Render();
    string GetStyle();
}

public class WindowsButton : IButton
{
    public void Render() => Console.WriteLine("Rendering Windows Button");
    public string GetStyle() => "Windows Style";
}

public class MacButton : IButton
{
    public void Render() => Console.WriteLine("Rendering Mac Button");
    public string GetStyle() => "Mac Style";
}

public class LinuxButton : IButton
{
    public void Render() => Console.WriteLine("Rendering Linux Button");
    public string GetStyle() => "Linux Style";
}

// ==================== 产品族 B：输入框 ====================

public interface ITextBox
{
    void Render();
    string GetStyle();
}

public class WindowsTextBox : ITextBox
{
    public void Render() => Console.WriteLine("Rendering Windows TextBox");
    public string GetStyle() => "Windows Style";
}

public class MacTextBox : ITextBox
{
    public void Render() => Console.WriteLine("Rendering Mac TextBox");
    public string GetStyle() => "Mac Style";
}

public class LinuxTextBox : ITextBox
{
    public void Render() => Console.WriteLine("Rendering Linux TextBox");
    public string GetStyle() => "Linux Style";
}

// ==================== 抽象工厂 ====================

/// <summary>
/// 抽象工厂：创建一族产品（同一主题下的所有 UI 组件）
/// </summary>
public interface IUIFactory
{
    IButton CreateButton();
    ITextBox CreateTextBox();
}

// ==================== 具体工厂 ====================

public class WindowsFactory : IUIFactory
{
    public IButton CreateButton() => new WindowsButton();
    public ITextBox CreateTextBox() => new WindowsTextBox();
}

public class MacFactory : IUIFactory
{
    public IButton CreateButton() => new MacButton();
    public ITextBox CreateTextBox() => new MacTextBox();
}

public class LinuxFactory : IUIFactory
{
    public IButton CreateButton() => new LinuxButton();
    public ITextBox CreateTextBox() => new LinuxTextBox();
}

// ==================== 客户端 ====================

/// <summary>
/// 客户端只依赖抽象，不依赖任何具体类
/// 切换平台只需注入不同的工厂
/// </summary>
public class Application
{
    private readonly IButton _button;
    private readonly ITextBox _textBox;

    public Application(IUIFactory factory)
    {
        _button = factory.CreateButton();
        _textBox = factory.CreateTextBox();
    }

    public void Render()
    {
        _button.Render();
        _textBox.Render();
        Console.WriteLine($"Style: {_button.GetStyle()}"); // 按钮和输入框风格一致
    }
}

class Program
{
    static void Main()
    {
        // 通过依赖注入，运行时决定使用哪套 UI 组件
        IUIFactory factory = new WindowsFactory();
        var app = new Application(factory);
        app.Render();
        // Rendering Windows Button
        // Rendering Windows TextBox
        // Style: Windows Style
    }
}
```

#### 抽象工厂的优缺点

| 优点 | 缺点 |
|------|------|
| 保证产品族一致性 | 新增产品族（新产品）需要修改所有工厂 |
| 隔离具体类的创建 | 新增产品（产品变体）需要添加新工厂 |
| 易于交换产品族 | 结构较复杂 |

### 3.4 Builder — 建造者模式

#### 模式意图

将一个复杂对象的**构建**与它的**表示**分离，使得同样的构建过程可以创建不同的表示。

#### 结构

```
Director      — 指挥者：控制构建顺序，使用 Builder 接口
     └─ Construct() → 调用 builder.BuildPartA() / BuildPartB() / ...

Builder       — 抽象建造者
     └─ BuildPartA()
     └─ BuildPartB()
     └─ GetResult() → Product

ConcreteBuilder — 具体建造者：实现各部件的构建方法，提供获取产品的方法
```

#### 核心问题解决

当一个对象构造参数过多（超过 3-4 个）、或者构造逻辑复杂分步骤、或者有些参数是可选的时候，构造函数会变得难以使用：

```csharp
// 构造函数地狱：7 个参数，且顺序不能错
var house = new House(200, 4, 2, true, false, "modern", "red");

// Builder 模式：清晰、可读、可选
var house = new HouseBuilder()
    .SetArea(200)
    .SetRooms(4)
    .SetBathrooms(2)
    .HasGarage(true)
    .HasSwimmingPool(false)
    .SetStyle("modern")
    .SetColor("red")
    .Build();
```

#### 代码实现

```csharp
// ==================== Product ====================

/// <summary>
/// 产品：计算机
/// </summary>
public class Computer
{
    public string CPU { get; set; }
    public string RAM { get; set; }
    public string Storage { get; set; }
    public string GPU { get; set; }
    public bool HasWiFi { get; set; }
    public bool HasBluetooth { get; set; }

    public override string ToString() =>
        $"CPU: {CPU}, RAM: {RAM}, Storage: {Storage}, GPU: {GPU}, WiFi: {HasWiFi}, Bluetooth: {HasBluetooth}";
}

// ==================== Builder ====================

/// <summary>
/// 抽象 Builder：定义构建部件的接口
/// </summary>
public interface IComputerBuilder
{
    IComputerBuilder SetCPU(string cpu);
    IComputerBuilder SetRAM(string ram);
    IComputerBuilder SetStorage(string storage);
    IComputerBuilder SetGPU(string gpu);
    IComputerBuilder SetWiFi(bool hasWiFi);
    IComputerBuilder SetBluetooth(bool hasBluetooth);
    Computer Build();
}

/// <summary>
/// 具体 Builder：实现计算机的构建细节
/// </summary>
public class ComputerBuilder : IComputerBuilder
{
    private Computer _computer = new Computer();

    public IComputerBuilder SetCPU(string cpu)
    {
        _computer.CPU = cpu;
        return this; // 链式调用
    }

    public IComputerBuilder SetRAM(string ram)
    {
        _computer.RAM = ram;
        return this;
    }

    public IComputerBuilder SetStorage(string storage)
    {
        _computer.Storage = storage;
        return this;
    }

    public IComputerBuilder SetGPU(string gpu)
    {
        _computer.GPU = gpu;
        return this;
    }

    public IComputerBuilder SetWiFi(bool hasWiFi)
    {
        _computer.HasWiFi = hasWiFi;
        return this;
    }

    public IComputerBuilder SetBluetooth(bool hasBluetooth)
    {
        _computer.HasBluetooth = hasBluetooth;
        return this;
    }

    public Computer Build()
    {
        // 可以在这里添加构建验证
        if (string.IsNullOrEmpty(_computer.CPU))
            throw new InvalidOperationException("CPU is required");
        if (string.IsNullOrEmpty(_computer.RAM))
            throw new InvalidOperationException("RAM is required");

        Computer result = _computer;
        _computer = new Computer(); // 重置Builder，支持复用
        return result;
    }
}

// ==================== Director ====================

/// <summary>
/// Director：封装构建步骤，提供常用的预设配置
/// </summary>
public class ComputerDirector
{
    /// <summary>
    /// 游戏电脑配置
    /// </summary>
    public Computer BuildGamingPC(IComputerBuilder builder)
    {
        return builder
            .SetCPU("Intel i9-13900K")
            .SetRAM("64GB DDR5")
            .SetStorage("2TB NVMe SSD")
            .SetGPU("RTX 4090")
            .SetWiFi(true)
            .SetBluetooth(true)
            .Build();
    }

    /// <summary>
    /// 办公电脑配置
    /// </summary>
    public Computer BuildOfficePC(IComputerBuilder builder)
    {
        return builder
            .SetCPU("Intel i5-13400")
            .SetRAM("16GB DDR4")
            .SetStorage("512GB SSD")
            .SetGPU("Integrated")
            .SetWiFi(true)
            .SetBluetooth(false)
            .Build();
    }

    /// <summary>
    /// 服务器配置
```

### 3.5 Prototype — 原型模式

#### 模式意图

通过**复制**现有对象（原型）来创建新对象，而无需知道其具体类型。适用于创建成本高（数据库查询、远程调用）的对象。

#### 核心概念：浅拷贝 vs 深拷贝

```
浅拷贝（Shallow Copy）
  ├─ 值类型字段：复制值
  └─ 引用类型字段：复制引用（与原对象共享同一引用）

深拷贝（Deep Copy）
  ├─ 值类型字段：复制值
  └─ 引用类型字段：递归复制整个对象（完全独立）
```

#### 代码实现

```csharp
// ==================== 支持原型的基类 ====================

/// <summary>
/// 原型接口：提供 Clone 方法
/// </summary>
public interface IPrototype<T>
{
    T Clone();
}

// ==================== 复杂对象：Employee ====================

public class Address
{
    public string City { get; set; }
    public string Street { get; set; }

    public Address(string city, string street)
    {
        City = city;
        Street = street;
    }

    // 深拷贝方法
    public Address DeepCopy() => new Address(City, Street);

    public override string ToString() => $"{City}, {Street}";
}

public class Employee : IPrototype<Employee>
{
    public string Name { get; set; }
    public int Age { get; set; }
    public Address Address { get; set; } // 引用类型字段
    public List<string> Skills { get; set; } // 引用类型字段

    public Employee(string name, int age, Address address, List<string> skills)
    {
        Name = name;
        Age = age;
        Address = address;
        Skills = skills;
    }

    // 浅拷贝：使用 C# 内置 MemberwiseClone
    // 引用类型字段（Address, List）复制的是引用，不是对象
    public Employee ShallowClone() => (Employee)MemberwiseClone();

    // 深拷贝：手动复制所有引用类型字段
    public Employee DeepClone()
    {
        return new Employee(
            Name,                  // string 是引用类型，但不可变，复制引用安全
            Age,                   // 值类型，直接复制
            Address.DeepCopy(),    // 递归深拷贝 Address
            new List<string>(Skills) // 创建新的 List 副本
        );
    }

    public override string ToString() =>
        $"Name: {Name}, Age: {Age}, Address: {Address}, Skills: [{string.Join(", ", Skills)}]";
}

// ==================== 测试代码 ====================

class Program
{
    static void Main()
    {
        var original = new Employee(
            "Alice",
            30,
            new Address("Beijing", "Zhangyang Rd"),
            new List<string> { "C#", "Python", "SQL" }
        );

        Console.WriteLine("=== Original ===");
        Console.WriteLine(original);
        // Name: Alice, Age: 30, Address: Beijing, Zhangyang Rd, Skills: [C#, Python, SQL]

        // 浅拷贝测试
        var shallowClone = original.ShallowClone();
        shallowClone.Name = "Bob";
        shallowClone.Address.City = "Shanghai"; // 影响原始对象！
        shallowClone.Skills.Add("Java");          // 影响原始对象！

        Console.WriteLine("\n=== After modifying ShallowClone ===");
        Console.WriteLine($"Original: {original}");
        // Name: Alice, Age: 30, Address: Shanghai, Zhangyang Rd, Skills: [C#, Python, SQL, Java]
        Console.WriteLine($"ShallowClone: {shallowClone}");
        // Name: Bob, Age: 30, Address: Shanghai, Zhangyang Rd, Skills: [C#, Python, SQL, Java]

        // 深拷贝测试
        var deepClone = original.DeepClone();
        deepClone.Name = "Charlie";
        deepClone.Address.City = "Guangzhou"; // 不影响原始对象！
        deepClone.Skills.Add("Go");           // 不影响原始对象！

        Console.WriteLine("\n=== After modifying DeepClone ===");
        Console.WriteLine($"Original: {original}");
        // Name: Alice, Age: 30, Address: Shanghai, Zhangyang Rd, Skills: [C#, Python, SQL, Java]
        Console.WriteLine($"DeepClone: {deepClone}");
        // Name: Charlie, Age: 30, Address: Guangzhou, Zhangyang Rd, Skills: [C#, Python, SQL, Java, Go]
    }
}
```

#### 深拷贝的替代方法

**序列化法（适用于支持序列化的对象）**

```csharp
public static class DeepCloneExtensions
{
    // 通过 JSON 序列化/反序列化实现深拷贝（简单但有性能开销）
    public static T DeepCloneViaJson<T>(this T obj)
    {
        string json = JsonSerializer.Serialize(obj);
        return JsonSerializer.Deserialize<T>(json)!;
    }
}

// 使用
var clone = original.DeepCloneViaJson();
```

#### 应用场景

- 对象创建成本高（数据库查询、网络请求、文件读取）
- 避免类的层次结构过于复杂（减少子类数量）
- 同一原型的不同实例（配置模板、游戏单位模板）

## 创建型模式对比总结

| 模式 | 核心思想 | 适用场景 | 复杂度 |
|------|---------|---------|--------|
| Singleton | 全局唯一实例 | 全局共享资源 | 低 |
| Factory Method | 子类决定创建 | 同类产品不同变体 | 中 |
| Abstract Factory | 创建产品族 | 系列产品一致性 | 中高 |
| Builder | 分步构建复杂对象 | 构造函数参数过多 | 中 |
| Prototype | 复制现有对象 | 创建成本高、对象相似 | 低中 |
