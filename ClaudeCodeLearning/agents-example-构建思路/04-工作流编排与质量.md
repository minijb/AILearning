# agents_example 插件分类分析 — 工作流、编排与质量

## 一、full-stack-orchestration — 全栈编排

### 1.1 插件概述

端到端全栈功能开发编排器，协调后端→前端→数据库→基础设施的多 Agent 工作流。

### 1.2 核心 Agent

| Agent | 职责 |
|-------|------|
| `backend-architect` | 后端 API 设计 |
| `database-architect` | 数据库架构 |
| `frontend-developer` | 前端开发 |
| `test-automator` | 测试自动化 |
| `security-auditor` | 安全审查 |
| `deployment-engineer` | 部署工程 |
| `observability-engineer` | 可观测性 |

### 1.3 全栈功能编排器（full-stack-feature）

**6 阶段工作流**：

| 阶段 | 内容 | 关键步骤 |
|------|------|---------|
| Phase 1 | 架构与设计基础 | 需求收集（交互式 Q&A） |
| Phase 2 | 后端实现 | API 端点、业务逻辑、数据模型 |
| Phase 3 | 前端实现 | UI 组件、状态管理、API 集成 |
| Phase 4 | 数据库和数据迁移 | Schema 迁移、种子数据 |
| Phase 5 | 测试与验证 | 单元/集成/E2E 测试 |
| Phase 6 | 部署和可观测性 | CI/CD、可观测性设置 |

### 1.4 状态文件

```json
// .full-stack-feature/state.json
{
  "feature": "$ARGUMENTS",
  "status": "in_progress",
  "stack": "auto-detect",
  "api_style": "rest",
  "complexity": "medium",
  "current_step": 1,
  "current_phase": 1,
  "completed_steps": [],
  "files_created": [],
  "started_at": "ISO_TIMESTAMP"
}
```

### 1.5 参数支持

```bash
/full-stack-feature "用户认证功能" \
  --stack react/fastapi/postgres \
  --api-style rest|graphql \
  --complexity simple|medium|complex
```

## 二、conductor — 上下文驱动开发

### 2.1 核心创新

Conductor 将 Claude Code 转变为**项目管理工具**，实现 Context → Spec → Plan → Implement 的结构化工作流。

### 2.2 与 full-stack-orchestration 的区别

| 维度 | full-stack-orchestration | conductor |
|------|--------------------------|-----------|
| **焦点** | 端到端功能交付 | 项目级上下文管理 |
| **时间跨度** | 单次功能开发 | 跨多会话持久化 |
| **产物** | 代码 + 测试 | 上下文 + 规格 + 计划 + 代码 |
| **用户控制** | 阶段门控确认 | Track 级别的语义撤销 |

### 2.3 Track（轨道）概念

```
Track = 功能/bug 的最小工作单位

conductor:new-track 创建：
  ├── spec.md        # 需求规格说明
  ├── plan.md        # 分阶段实施计划
  ├── metadata.json   # 元数据
  └── index.md        # Track 导航
```

### 2.4 语义撤销（Semantic Revert）

Conductor 支持**按逻辑单元撤销**（而非 git 的文件级撤销）：

```
conductor:revert --track <track-id>    # 撤销整个 track
conductor:revert --phase <phase-num>   # 撤销特定阶段
conductor:revert --task <task-id>       # 撤销特定任务
```

这依赖 Git 的实现来跟踪逻辑单元。

## 十三、core 设计亮点

### 13.1 阶段门控的 UX 设计

full-stack-orchestration 和 security-scanning 都使用 `PHASE CHECKPOINT`：
- 在进入下一阶段前**强制用户确认**
- 用户可以选择：继续 / 修改计划 / 终止
- 这使 AI 的自主性和人类控制达到平衡

### 13.2 状态文件作为"记忆"

`.security-hardening/state.json` 和 `.full-stack-feature/state.json` 解决了 AI 会话的"健忘"问题：
- 即使上下文压缩，状态文件仍然存在
- 新会话可以恢复进度
- `started_at` 和 `last_updated` 追踪时间线

### 13.3 Conductor 的 Track 粒度

Track 是比 git commit 更高级的工作单元：
- 一个 Track = 一个 spec.md + 一个 plan.md + 多个 phase
- 一个 Phase = 多个 Task
- 语义撤销（`--track`）比 git revert 更直观
