# Claude Code Plugins — agents_example 构建思路总览

## 项目概述

agents_example（GitHub: wshobson/agents）是 GitHub 上另一个极具影响力的 Claude Code 插件集合，荣膺 Anthropic 官方精选。它采用**原子化插件市场**架构，将功能拆分为极细粒度的独立插件，每个插件平均仅 3.4 个组件，践行 Unix 哲学的"做一件事并做好"。

**核心数据**：
- **74 个独立插件**，覆盖 24 个领域类别
- **112 个专业 Agent**
- **129 个 Skill**（遵循渐进式披露架构）
- **79 个开发工具**（Commands）
- **PluginEval**：三层质量评估框架

> 插件市场地址：https://github.com/wshobson/agents
> 安装：`/plugin marketplace add wshobson/agents`

## 一、核心设计哲学

### 1.1 原子化插件架构（Granular Plugin Architecture）

与 everything-claude-code 的"大而全"设计不同，agents_example 追求极致的**单一职责**：

| 设计原则 | 说明 |
|---------|------|
| Single Responsibility | 每个插件专注一件事（5-10 词描述） |
| 平均组件数 3.4 | 遵循 Anthropic 的 2-8 组件模式 |
| 零膨胀 | 不做大而笨的插件 |
| 可组合性 | 按需混搭，不强制捆绑 |

**安装对比**：
```bash
# agents_example：只加载需要的
/plugin install python-development  # 仅加载 3 个 Python Agent + 1 个工具 + 5 个 Skill ≈ 1000 tokens

# everything-claude-code：全量加载
# 30 Agents + 135 Skills + 60 Commands ≈ 更多 tokens
```

### 1.2 渐进式披露（Progressive Disclosure）

Skills 采用三层架构，**只在需要时加载**对应层级的知识：

```
Tier 1 - Metadata（始终加载）
  └── name + description（激活条件）

Tier 2 - Instructions（激活时加载）
  └── 核心指导和模式

Tier 3 - Resources（按需加载）
  └── 示例和模板
```

这使 Token 使用量精确可控——不需要全量加载所有知识。

### 1.3 三层模型策略（Three-Tier Model Strategy）

| 层级 | 模型 | Agent 数量 | 适用场景 |
|------|------|-----------|---------|
| **Tier 1** | Opus 4.6 | 42 | 关键架构、安全、所有代码审查、生产编码 |
| **Tier 2** | Inherit | 42 | 复杂任务，用户选择模型（AI/ML、后端、前端） |
| **Tier 3** | Sonnet | 51 | 支持性任务（文档、测试、调试、网络、API 文档） |
| **Tier 4** | Haiku | 18 | 快速操作任务（SEO、部署、简单文档、销售、内容） |

**核心洞察**：
- Opus 的 65% Token 节省通常抵消了较高的单价
- `inherit` 层级允许用户在特定场景选择成本与能力的平衡

## 二、PluginEval — 质量评估框架

PluginEval 是 agents_example 最具创新的组件之一，提供**三层评估体系**：

### 2.1 三层评估架构

| 层级 | 方法 | 时间 | 成本 |
|------|------|------|------|
| **Layer 1: Static** | 结构化分析、反模式检测 | < 2 秒 | 免费 |
| **Layer 2: LLM Judge** | 语义评估（触发、准确性、编排、输出） | ~30 秒 | 4 次 LLM 调用 |
| **Layer 3: Monte Carlo** | 50-100 次模拟运行的统计可靠性 | ~2-5 分钟 | 50-100 次调用 |

### 2.2 十大质量维度

| 维度 | 权重 | 说明 |
|------|------|------|
| triggering_accuracy | 25% | 触发条件匹配度 |
| orchestration_fitness | 20% | 工作流适配度 |
| output_quality | 15% | 输出质量 |
| scope_calibration | 12% | 范围校准 |
| progressive_disclosure | 10% | 渐进式披露 |
| token_efficiency | 6% | Token 效率 |
| robustness | 5% | 健壮性 |
| structural_completeness | 3% | 结构完整性 |
| code_template_quality | 2% | 代码模板质量 |
| ecosystem_coherence | 2% | 生态一致性 |

### 2.3 质量徽章

- **Platinum ≥90**：★★★★★
- **Gold ≥80**：★★★★
- **Silver ≥70**：★★★
- **Bronze ≥60**：★★

### 2.4 反模式检测

| 反模式 | 描述 |
|--------|------|
| OVER_CONSTRAINED | 超过 15 个 MUST/ALWAYS/NEVER |
| EMPTY_DESCRIPTION | 描述 < 20 字符 |
| MISSING_TRIGGER | 无 "Use when..." 触发条件 |
| BLOATED_SKILL | 无引用的 > 800 行 |
| ORPHAN_REFERENCE | 死链接 |
| DEAD_CROSS_REF | 缺失的 skill 交叉引用 |

## 三、Agent Teams — 多 Agent 编排

Agent Teams 是 Claude Code 的实验性功能，用于**并行多 Agent 工作流**。

### 3.1 团队角色

| Agent | 角色 | 职责 |
|-------|------|------|
| `team-lead` | 团队协调者 | 分解工作、管理生命周期、综合结果 |
| `team-reviewer` | 多维审查者 | 按分配的维度进行代码审查 |
| `team-debugger` | 假设调查者 | 收集证据确认/否定假设 |
| `team-implementer` | 并行构建者 | 在严格文件所有权边界内实现 |

### 3.2 七支预设团队

| 预设 | 用途 | 特点 |
|------|------|------|
| `review` | 多维并行审查 | 安全+性能+架构+可访问性 |
| `debug` | 假设驱动调试 | 多假设并行调查 |
| `feature` | 并行功能开发 | 文件所有权边界 |
| `fullstack` | 全栈协调 | 后端+前端并行 |
| `research` | 并行研究 | 多探索代理同时工作 |
| `security` | 安全审计 | OWASP+认证+依赖+配置 |
| `migration` | 迁移协调 | 并行流+正确性验证 |

### 3.3 关键命令

| 命令 | 功能 |
|------|------|
| `/team-spawn` | 使用预设或自定义组合生成团队 |
| `/team-status` | 显示团队成员、任务和进度 |
| `/team-review` | 多审查者并行代码审查 |
| `/team-debug` | 多假设并行调试 |
| `/team-feature` | 带文件所有权的并行功能开发 |

## 四、Conductor — 上下文驱动开发

Conductor 将 Claude Code 转变为**项目管理工具**，实现 Context → Spec → Plan → Implement 的结构化工作流。

### 4.1 核心命令

| 命令 | 功能 |
|------|------|
| `/conductor:setup` | 初始化项目（产品定义、技术栈、工作流、风格指南） |
| `/conductor:new-track` | 创建功能/bug track（含 spec.md 和 plan.md） |
| `/conductor:implement` | 按计划执行任务（含验证检查点） |
| `/conductor:status` | 显示项目进度概览 |
| `/conductor:revert` | Git 感知的撤销（按 track/phase/task） |

### 4.2 生成的结构化产物

```
conductor/
├── index.md              # 导航中心
├── product.md            # 产品愿景和目标
├── product-guidelines.md # 标准和消息规范
├── tech-stack.md         # 技术偏好
├── workflow.md          # 开发实践（TDD、提交规范）
├── tracks.md            # Track 注册表
└── tracks/
    └── <track-id>/
        ├── spec.md       # 需求规格
        ├── plan.md       # 分阶段任务分解
        ├── metadata.json # Track 元数据
        └── index.md      # Track 导航
```

### 4.3 TDD 工作流集成

```
conductor:implement 执行时遵循 TDD 循环：
  RED: 写一个会失败的测试
  GREEN: 写最少量代码让测试通过
  REFACTOR: 重构，测试保持绿色
```

## 六、代表性插件深度分析

### 6.1 full-stack-orchestration — 全栈编排器

**设计亮点**：阶段门控（Phase Checkpoint）模式

```markdown
## CRITICAL BEHAVIORAL RULES
1. Execute steps in order. Do NOT skip ahead.
2. Write output files. Each step MUST produce its output file.
3. Stop at PHASE CHECKPOINT. MUST wait for user approval.
4. Halt on failure. STOP immediately on error.
5. Use only local agents. No cross-plugin dependencies.
```

**工作流**：
```
Phase 1: 架构与设计基础（交互式需求收集）
Phase 2: 后端实现
Phase 3: 前端实现
Phase 4: 数据库和数据迁移
Phase 5: 测试与验证
Phase 6: 部署和可观测性
```

### 6.2 security-scanning — 安全扫描

**设计亮点**：多阶段深度安全评估

```markdown
Phase 1: 评估与威胁建模（漏洞扫描）
Phase 2: SAST 静态分析
Phase 3: 依赖扫描
Phase 4: 代码审查（认证、授权、加密、注入）
Phase 5: 配置审计
Phase 6: 报告生成
```

支持合规框架：OWASP、SOC2、GDPR、HIPAA、PCIDSS

### 6.3 kubernetes-operations — K8s 操作

**包含 4 个 Skill**：
- k8s-manifest-generator：生产级 K8s YAML 生成
- helm-chart-scaffolding：Helm chart 模板化管理
- gitops-workflow：ArgoCD/Flux 自动化部署
- k8s-security-policies：NetworkPolicy、PodSecurityPolicy、RBAC

### 6.4 llm-application-dev — LLM 应用开发

**8 个专业 Skill**，覆盖 LangChain/LangGraph、RAG、Prompt 工程、Embedding、评估等：

| Skill | 描述 |
|-------|------|
| langchain-architecture | LangGraph agent 架构（2026 标准） |
| rag-implementation | 向量数据库 + 语义搜索 RAG |
| prompt-engineering-patterns | Prompt 性能优化 |
| llm-evaluation | 自动化评估策略 |

## 七、Agent 文件格式规范

### 7.1 Frontmatter 格式

```yaml
---
name: backend-architect              # 唯一标识符
description: "专家后端架构师... Use PROACTIVELY when [触发条件]."
model: inherit                        # opus/sonnet/haiku/inherit
color: blue                          # 可选：UI 颜色
---
```

**关键设计**：
- `description` 中的 "Use PROACTIVELY when" 是**主动调度信号**
- `model: inherit` 表示使用会话默认模型（Tier 2）

### 7.2 Agent 与 Skill 的协同模式

```
用户请求: "为文档 Q&A 构建 RAG 系统"
    ↓
langchain-architecture skill → LangGraph 架构指导
    ↓
rag-implementation skill → RAG 实现模式
    ↓
embedding-strategies skill → Embedding 管道设计
    ↓
llm-evaluation skill → 评估策略
```

Agent 负责高层推理和编排，Skill 提供具体领域知识。

## 八、Skill 文件格式规范

### 8.1 Frontmatter

```yaml
---
name: api-design-principles              # hyphen-case
description: "Master REST and GraphQL API design... Use when [触发条件]."
version: 1.0.0                           # 可选
---
```

### 8.2 内部结构

```
skills/<skill-name>/
├── SKILL.md                 # 必需：frontmatter + 内容
├── references/              # 可选：支撑材料
│   └── *.md
└── assets/                 # 可选：模板、配置
    └── *.md
```

### 8.3 "Use when" 触发模式

Skills 的 description 必须包含 "Use when" 子句，用于自动激活：

```yaml
description: "Master REST and GraphQL API design...
Use when designing new APIs, reviewing API specifications,
or establishing API design standards."
```

## 十、与 everything-claude-code 的设计对比

| 维度 | agents_example | everything-claude-code |
|------|--------------|----------------------|
| **插件数量** | 74 个原子化插件 | 1 个大插件 |
| **设计哲学** | 原子化 + 可组合 | 大而全 + 全量加载 |
| **平均组件/插件** | 3.4 | ~200+ |
| **Skill 数量** | 129（渐进式披露） | 135+ |
| **质量框架** | PluginEval（三层评估） | 无正式评估框架 |
| **多 Agent 编排** | Agent Teams（实验性） | /orchestrate（命令行） |
| **上下文管理** | Conductor（项目管理） | Hooks（自动化） |
| **Hook 系统** | 无 | 完整的 PreToolUse/PostToolUse 链 |
| **持续学习** | 无 | 本能（Instinct）架构 |
| **学习来源** | 手工编写 | Hook 自动捕获 |
| **Token 策略** | 渐进式披露 + 原子化 | 全量加载 + 上下文动态注入 |
| **模型分层** | 4 层（Opus/Sonnet/Haiku/Inherit） | 3 层（Opus/Sonnet/Haiku） |

**互补性**：两个项目代表了两种截然不同的设计哲学，可以互补使用：
- agents_example：轻量级按需插件 + 高质量 Skill 库
- everything-claude-code：完整 Hook 系统 + 持续学习 + 全量自动化
