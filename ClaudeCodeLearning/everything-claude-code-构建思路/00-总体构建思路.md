# Everything Claude Code — 总体构建思路

## 项目概述

Everything Claude Code（ECC）是 GitHub 上最知名的 Claude Code 插件之一，荣膺 Anthropic x Forum Ventures 黑客松冠军。它是一套**生产级别的 AI 编码工作流系统**，包含 30 个专家 Agent、135+ 工作流 Skill、60 个斜杠命令，以及完整的 Hook 生命周期管理体系。

> 项目地址：https://github.com/affaan-m/everything-claude-code
> 核心哲学文档：SOUL.md

---

## 一、核心设计哲学（SOUL.md）

ECC 的灵魂由五大原则构成，所有组件都围绕这些原则展开：

| 原则 | 含义 | 落地方式 |
|------|------|----------|
| **Agent-First** | 尽早将工作路由到正确的专家 Agent | 30 个专业 Agent，覆盖审查/规划/构建/测试全流程 |
| **Test-Driven** | 写测试优先于信任实现 | TDD Guide Agent + /tdd 命令强制执行 RED→GREEN→REFACTOR 循环 |
| **Security-First** | 验证输入、保护秘密、安全默认值 | Security Reviewer Agent + 安全 Rules + 秘密检测 Hook |
| **Immutability** | 优先显式状态转换而非修改 | Coding Style Rules 强制不可变数据结构 |
| **Plan Before Execute** | 复杂变更分解为刻意阶段 | /plan、/orchestrate 等命令支持阶段性规划 |

## 三、各子系统定位

### 3.1 Agents — 专家子代理层

**作用**：将复杂任务委托给专门的子代理，每个 Agent 有明确的职责边界和置信度阈值。

- **设计亮点**：
  - YAML frontmatter 元数据（`name`/`description`/`tools`/`model`）
  - `description` 字段用于**自动调度决策**
  - 80%+ 置信度才报告问题（避免噪音）
  - 严重性分级：CRITICAL > HIGH > MEDIUM > LOW
  - 输出格式标准化（Summary Table + Verdict）

**分类**：
| 类型 | 代表 Agent | 职责 |
|------|-----------|------|
| 审查类 | code-reviewer, security-reviewer, tdd-guide | 质量/安全/TDD 审查 |
| 构建类 | build-error-resolver, *-build-resolver (8种) | 工具链错误修复 |
| 语言专项 | go-reviewer, rust-reviewer, cpp-reviewer 等 | 特定语言审查 |
| 运营类 | loop-operator, harness-optimizer | 自主循环监控 |
| 文档类 | doc-updater, docs-lookup | 文档同步/查询 |
| 通信类 | chief-of-staff | 跨渠道通信管理 |

### 3.2 Skills — 工作流与领域知识

**作用**：定义 Claude 在特定场景下的行为模式和工作流程。

**分类**：
- **行为类**：`coding-standards/`、`tdd-workflow/`、`security-review/`、`backend-patterns/`、`frontend-patterns/`
- **学习系统**：`continuous-learning-v2/` — 本能（Instinct）架构，原子级学习单位 + 置信度评分

**设计亮点**：
- Markdown 格式，结构化 sections
- 每个 Skill 可独立激活或自动触发
- 置信度 0.3–0.9，影响应用优先级

### 3.3 Commands — 斜杠命令层

**作用**：用户主动调用的快捷工作流入口，是 Skills 的快速执行版本。

**代表命令**：
| 命令 | 功能 |
|------|------|
| `/tdd` | 完整 TDD 流程（RED→GREEN→REFACTOR） |
| `/plan` | 阶段性规划（等待用户确认后才开始） |
| `/orchestrate` | 多 Agent 编排（planner→tdd-guide→reviewer→security） |
| `/model-route` | 模型选择建议（Haiku/Sonnet/Opus） |
| `/skill-create` | 从 Git 历史提取模式生成 Skill |
| `/evolve` | 将本能演化为完整 Skill/Command/Agent |
| `/learn` | 从当前会话提取可重用模式 |
| `/loop-start` | 启动托管自主循环（safe/fast 模式） |

### 3.4 Rules — 始终遵循的指南

**作用**：Claude 在所有会话中都会遵守的规则，通过 `.claude/rules/` 目录实现。

**结构**：
```
rules/
├── common/          # 通用规则（所有语言）
│   ├── coding-style.md      # CRITICAL: 不可变性、小文件原则(200-400行)
│   ├── testing.md           # 80%覆盖率底线、TDD强制流程
│   ├── security.md          # 安全检查表、秘密管理
│   ├── agents.md             # 并行执行原则、Agent编排策略
│   ├── performance.md        # 模型选择、上下文管理
│   ├── patterns.md           # Repository模式、API响应格式
│   ├── git-workflow.md      # 提交格式、PR流程
│   ├── hooks.md             # Hook类型、TodoWrite最佳实践
│   └── code-review.md       # 审查流程
├── typescript/      # TypeScript规则（无any、Zod验证等）
├── python/          # Python规则
├── golang/         # Go规则
├── rust/           # Rust规则
├── cpp/            # C++规则
├── zh/             # 中文规则集
└── ...             # 共11种语言
```

### 3.5 Hooks — 事件驱动自动化

**作用**：在 Claude Code 生命周期关键节点自动执行脚本，无需用户介入。

**完整 Hook 链**：
| 阶段 | 触发时机 | 功能 |
|------|---------|------|
| **PreToolUse** | 工具执行前 | 安全检查（禁止 --no-verify）、tmux 提醒、格式化建议、console.log 检查 |
| **PostToolUse** | 工具执行后 | 自动格式化（Biome/Prettier）、类型检查、console.warn、质量门禁 |
| **PreCompact** | 上下文压缩前 | 保存状态到文件 |
| **SessionStart** | 新会话开始 | 加载前序上下文、检测包管理器 |
| **Stop** | 响应结束时 | console.log 检查、session 持久化、会话评估、成本追踪、桌面通知 |
| **SessionEnd** | 会话真正结束时 | 生命周期标记 |

**设计亮点**：
- `run-with-flags.js` 统一封装，支持 `ECC_HOOK_PROFILE` 和 `ECC_DISABLED_HOOKS` 运行时门控
- 异步 Hook（`async: true`）不阻塞主流程
- 所有 Hook 在解析错误时必须 `exit 0`，避免阻断工具执行

### 3.6 Contexts — 动态系统提示

**作用**：通过 CLI 标志动态注入不同场景下的系统提示，而非全部加载。

**三种模式**：
| 文件 | 模式 | 核心理念 |
|------|------|---------|
| `dev.md` | 开发模式 | 先让它工作→再让它正确→最后让它干净 |
| `review.md` | 审查模式 | 按严重性分级，建议修复而非仅指出问题 |
| `research.md` | 研究模式 | 广泛阅读后再下结论，按需使用 WebSearch |

**使用方式**：
```bash
# 开发模式
alias claude-dev='claude --system-prompt "$(cat ~/.claude/contexts/dev.md)"'

# 审查模式
alias claude-review='claude --system-prompt "$(cat ~/.claude/contexts/review.md)"'
```

### 3.7 MCP Configs — 外部服务集成

**作用**：通过 Model Context Protocol 连接外部服务（GitHub、Supabase、Vercel 等）。

**配置策略**：
- 在配置文件中声明所有 MCP，但通过 `disabledMcpServers` 按项目禁用不需要的
- **原则**：保持 10 个以下启用，MCP 会显著占用上下文窗口

## 四、核心工作流设计

### 4.1 Feature 开发工作流

```
用户请求
    ↓
/plan 命令 → Planner Agent（分阶段规划）
    ↓
/orchestrate feature → 多 Agent 串联
    ↓
    ├→ TDD Guide Agent（RED→GREEN→REFACTOR）
    ├→ Code Reviewer Agent（质量审查）
    ├→ Security Reviewer Agent（安全审查）
    └→ Architect Agent（架构验证）
    ↓
质量门禁（Hooks: 格式化+类型检查+console.log检查）
    ↓
PR 创建（Hooks: git push 提醒+提交质量检查）
```

### 4.2 持续学习闭环（Continuous Learning v2.1）

```
会话活动（Hooks 捕获 100% 可靠的工具调用）
    ↓
项目作用域隔离（git remote URL 哈希 → 项目 ID）
    ↓
观察数据 → projects/<hash>/observations.jsonl
    ↓
Haiku 后台 Agent 分析 → 模式检测
    ↓
本能（Instinct）生成 → 置信度 0.3–0.9
    ↓
/evolve 聚类 → Skill / Command / Agent
    ↓
/promote → 项目本能 → 全局本能（跨项目验证后）
```

## 五、关键设计模式总结

### 5.1 分层职责分离

```
用户 ← Commands（主动触发）
         ↓
     Skills（自动激活/场景匹配）
         ↓
     Agents（子代理深度执行）
         ↓
     Rules（始终约束）
         ↓
     Hooks（生命周期自动化）
```

### 5.2 Token 经济优化

- **模型分层**：Haiku（90% 能力/3x 节省）用于轻量任务，Sonnet 作为主力，Opus 用于复杂架构
- **上下文隔离**：Contexts 按模式动态注入，不全量加载
- **MCP 精简**：配置多、启用少（<10 个活跃）
- **mgrep > grep**：约 50% Token 节省

### 5.3 可靠性工程

- **Hook > Prompt**：Hook 以 100% 确定性触发，Prompt 概率性触发（LLM 约 20% 遗忘）
- **退出码原则**：所有 Hook 在非关键错误时必须 `exit 0`
- **置信度阈值**：Agent 仅在 ≥80% 置信度时才报告问题
- **停止条件优先**：循环操作必须有明确停止条件（`loop-operator`）

## 七、构建思路核心启示

ECC 的构建哲学可以归纳为以下可复用的设计原则：

1. **Agent-First 编排**：不是让一个通用 Agent 做所有事，而是尽早路由到专家 Agent，每个 Agent 职责单一
2. **确定性优于概率性**：Hook 的 100% 触发率 > Skill 的概率触发；脚本的确定性逻辑 > LLM 的推理
3. **原子化学习**：不直接生成 Skill，而是先生成带置信度的"本能"，再聚类演进为完整组件
4. **Token 经济意识**：从模型选择、上下文管理、MCP 配置到文件大小，处处考虑 Token 成本
5. **渐进式复杂性**：简单任务用 Haiku + Hook；复杂任务用 Sonnet + Orchestrate；架构决策用 Opus + Architect
6. **项目隔离**：本能系统 v2.1 通过项目作用域隔离，避免跨项目知识污染
7. **会话持久化**：PreCompact + Stop Hook 构建跨会话记忆，使 Claude 能"接着上次继续"
